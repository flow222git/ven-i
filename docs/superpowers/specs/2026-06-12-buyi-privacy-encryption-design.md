# 私密書寫類工具 隱私與安全感設計

**日期：** 2026-06-12
**狀態：** 設計定案，待寫實作計畫
**範圍：** 平台上所有含「私密自由書寫」的工具 —— 首發 ven-i（卜易／問易）與**心理位移書寫**；同步影響 Firestore 資料結構與 admin 後台
**定位：** 本設計是**平台層級的共用隱私原則**，不是單一工具的特例。任何讓使用者寫下私密內容的工具都比照辦理：**管理員（營運方）不應看到任何內容**。

---

## 1. 問題與目標

卜易／問易的使用者拿來問的常是感情、金錢、健康、恐懼這類最私密的事。目前 ven-i 把使用者的記錄以**明文**存進 `Firestore users/{email}/daily/{date}`，而平台另有 `admin.html` 後台本就設計成能看「全部記錄／使用者詳情」。也就是說，營運方目前**技術上看得到每個人卜了什麼、寫了什麼筆記**。這會讓使用者擔心隱私，傷害工具的安全感。

**目標：**讓「使用者的私密內容不會被任何人（含營運方）看到」成為**技術上的保證**，而非只是口頭承諾；同時**完全不犧牲**營運方想要的後台大數據分析。

**營運方真正需要的後台資料（使用者已確認）：** 被問的「類型」、某期間哪些卦最常出現、多少人在用、使用趨勢。皆為統計／彙總，**不需要**任何私密內容原文。

---

## 2. 核心設計：把資料拆成兩層

使用者想要的分析與使用者的隱私其實不衝突 —— 它們是兩種不同性質的資料，目前只是被混在同一筆記錄裡存。拆開即可兩全。**此兩層模型是所有私密書寫工具的共同骨架**，差別只在「可分析層」放哪些欄位、「私密層」放哪些欄位（見 §2.3 各工具對應）。

### 📊 可分析層（明文、上雲、後台看得到）
- `hexagram`：本卦／變卦（卦象）→ 支援「哪些卦最常出現」
- `category`：類型標籤，列舉值（感情／事業／健康／決定／其他）→ 支援「被問的類型」
  - **重點：`category` 是使用者卜卦時自己點選的下拉分類，不是去讀問題原文猜出來的。** 後台拿到「類型」分布，但讀不到任何一句問題。
- `ts`：時間
- `source`：工具來源（沿用現有欄位）
- 使用者數、活躍度等由上述彙總得出

### 🔒 私密層（使用者的自由文字）
- `question`：問題原文（使用者實際打的那句話）
- `notes`：筆記

私密層預設不上雲；使用者主動開啟跨裝置同步時，以信封加密後才上雲，存進 Firestore 的永遠是亂碼。

### 2.3 各工具的欄位對應
同一套兩層模型、同一套加密機制（§3、§4），各工具只是欄位歸屬不同：

**卜易／問易**
- 可分析層：`hexagram`（本卦／變卦）、`category`（類型標籤，使用者自選）、`ts`、`source`
- 私密層：`question`（問題原文）、`notes`（筆記）

**心理位移書寫**
- 內容幾乎全屬私密 —— 書寫本文就是最私密的東西，**管理員一律不可見**。
- 可分析層（只有純使用指標，不含任何文字）：`ts`、`source`、是否完成各人稱階段的旗標（如「我／你／他」三段是否寫完）、字數等量化指標。
- 私密層：三種人稱的書寫本文全部內容。
- 設計重點：心理位移沒有「類型／卦象」這種可彙總的結構欄位，因此可分析層**僅止於使用量與完成度**；後台只看得到「有人寫了、寫了幾段、何時寫」，看不到一個字。

## 3. 私密層的存放策略：B2 信封加密（漸進式啟用）

### 3.1 漸進式啟用（降低摩擦）
- **預設（未設密語）：** 私密層只存使用者自己的瀏覽器（localStorage / IndexedDB），**完全不上傳**。一般休閒使用者一輩子不會被密碼流程打擾，私密內容也天然安全（不在伺服器上）。
- **使用者主動開啟「跨裝置私密同步」時：** 才請他設一組通關密語，升級成信封加密上雲。
- 可分析層**自始至終都同步**，因此日曆時間軸、後台大數據從第一天就完整，不受私密層策略影響。

### 3.2 信封加密（envelope encryption）
真 E2E 的鐵則：加密金鑰必須是**只有使用者握著、伺服器拿不到也產不出**的祕密，否則營運方（或被駭、被法院要求時）就能解。Google 登入只證明身分，不提供這把鑰匙，因此必須有一個**使用者自設、明文絕不上傳**的祕密。

機制：
1. **DEK（資料金鑰）：** 啟用時於瀏覽器內隨機產生一把 256-bit AES 金鑰，真正用來加密私密欄位的就是它。
2. **用通關密語包住 DEK：** 由密語經 KDF 導出一把 wrapping key，把 DEK 加密（wrap）後，只把**密文**存進 Firestore。
3. **用恢復碼也包一份 DEK：** 另產生一張隨機恢復碼，同樣導出一把 key 把 DEK 再 wrap 一份存上去。（見 §4 忘記密語後路）
4. **加密私密欄位：** `question`／`notes` 用 DEK（AES-GCM、每筆隨機 IV）加密後存。
5. Firestore 中：可分析層明文、私密層密文、兩份 wrapped DEK、salt、verifier。**伺服器全程只見亂碼與包好的金鑰，解不開。**

**一把 DEK 管全平台：** 因平台已是同網域 SSO（見 [[jte-sso-userbar]]），每位使用者用**單一 DEK／單一通關密語**加密其在所有私密書寫工具（卜易、心理位移…）的私密內容。使用者只需設一次密語、記一張恢復碼，即可解鎖全平台自己的私密資料；wrap 後的金鑰存在使用者層（`users/{email}`）而非各工具下。

### 3.3 密碼學選型（皆瀏覽器內建 WebCrypto / SubtleCrypto，GitHub Pages HTTPS 可用）
- 對稱加密：**AES-GCM 256**，每筆私密記錄獨立隨機 96-bit IV。
- 金鑰導出：**PBKDF2**（高 iteration count）+ 每使用者隨機 salt（salt 非機密，可明文存 Firestore）。
- DEK wrap：以導出的 key 對 DEK 做 AES-GCM 加密。
- 密語驗證：另存一個用 DEK 加密的已知 verifier 值，解鎖時用來判斷密語是否正確（避免拿錯密語默默解出亂碼）。

## 4. 忘記密語的後路：密語 ＋ 備援恢復碼（選定方案）

真 E2E 唯一逃不掉的代價是「營運方救不了使用者，因為本來就沒鑰匙」。本工具的使用者輪廓是**久久用一次、內容又很有感情**，忘記密語是**預期常態**而非邊緣案例；若有人弄丟一年前放在心上的筆記，崩潰感正好與「安心」相反。因此提供恢復碼作為安全網。

- **設定時（一次）：** 輸入通關密語 → 系統另給一張恢復碼，提供「下載備份」，旁註「忘記密語時靠這張救回，我們沒有副本」。
- **換裝置：** 輸入密語即解鎖。
- **忘記密語：** 點「用恢復碼重設」→ 貼上恢復碼解出 DEK → 設定新密語（重新 wrap）。
- **兩者都遺失：** 才真正無法救回（這是真 E2E 不可避免的底線）。

實作上即把同一把 DEK **包兩次**（密語一份、恢復碼一份），E2E 保證不打折。

## 5. 使用者流程（UX）

1. **啟用跨裝置私密同步：** 設定密語（輸入兩次確認）→ 顯示／下載恢復碼 → 完成。
2. **同裝置續用：** 解鎖後把 DEK 暫存本機（標記為信任裝置），之後免重打。
3. **新裝置解鎖：** Google 登入後，提示「輸入通關密語解鎖私密內容」；可勾「在這台裝置記住」。
4. **忘記密語：** 「用恢復碼重設」流程（§4）。
5. **未啟用者：** 私密層僅存本機，無任何密碼互動。

### 信任感的可見化（無論啟用與否都要做）
- 私密欄位旁的鎖頭圖示與「只有你看得到」說明文案。
- 一頁簡明隱私說明（實際文案見 §5.1）。
- 啟用加密後顯示「已加密同步，連我們也解不開」狀態。

### 5.1 給使用者看的隱私聲明（文案草稿）
放在私密書寫入口、設定頁與「了解隱私」連結處，用白話讓使用者一眼安心：

> **🔒 你的私密書寫，只有你看得到 —— 這是技術保證，不只是承諾**
>
> 我們知道你在這裡寫下的，常是最私密的心事。所以我們從設計上就讓自己「看不到」：
>
> **只有你能看到的（我們完全看不到）**
> - 你卜卦時打下的問題、你的筆記
> - 你的心理位移書寫全文
> 這些內容在離開你的裝置前，就已經在你自己的瀏覽器裡用你的通關密語加密。上傳到雲端的是一段亂碼，**連我們也解不開**。
>
> **我們看得到的（只有統計，沒有你的內容）**
> - 卜到的卦象、你自己點選的問題類型、時間
> - 有多少人在用、哪些卦較常出現這類整體趨勢
> 這些幫助我們把工具做得更好，但其中**沒有任何一句你寫的話**。
>
> **關於你的通關密語**
> 它只存在你心裡和你的裝置，我們手上沒有任何副本。這正是「連我們也解不開」的原因。也因此，請務必保存好設定時給你的**恢復碼** —— 萬一忘記密語，它是唯一能救回你內容的鑰匙；兩者都遺失，我們也無法幫你救回。
>
> 你可以放心地寫。

## 6. 對 admin 後台的影響
- 後台分析改為**只讀可分析層**（卦象、`category`、時間、來源、計數），產出類型分布、熱門卦、使用量／趨勢。
- 後台**不再顯示**任何私密欄位；對已加密記錄，私密欄位本就是亂碼。
- 「全部記錄／使用者詳情」頁需移除私密內容欄位，改呈現彙總或可分析層。

## 7. 資料結構變更（概念）
現行每日 `linked[]` 內每筆記錄需從「混合明文」改為分層：
```
{
  source, ts,
  hexagram, category,           // 可分析層：明文
  private: {                    // 私密層
    // 未啟用加密：明文，僅存本機、不上傳
    // 已啟用加密：{ iv, ciphertext } 上傳 Firestore
    question, notes
  }
}
```
使用者層另存一次：`salt`、`wrappedDEK_passphrase`、`wrappedDEK_recovery`、`verifier`（皆非機密或為密文，可放 Firestore `users/{email}` 下）。

## 9. 測試考量
- 加解密 round-trip：同密語可解、錯密語拒解（verifier 生效）。
- 恢復碼路徑：用恢復碼解出 DEK 並成功重設密語。
- 跨裝置：A 裝置加密、B 裝置輸入密語可讀。
- 後台：確認任何視圖都拿不到私密原文（含已加密記錄顯示為亂碼）。
- 漸進式：未啟用時私密層不出現在任何上傳 payload。
